# ADR-004: HTTP Basic Auth (No OAuth2)

**Status:** Accepted

**Date:** 2026-06-01

## Context

The platform is private (VPN-only access) with a single operator user. No user registration, SSO, or session management is required. The deployment is behind nginx with `allow 10.0.0.0/24; deny all;`.

## Decision

Use HTTP Basic Authentication, verified in application middleware (not nginx). Credentials are configured via environment variables `DASH_USER`/`DASH_PASS` or `.credentials.json` file (chmod 0600). Comparison uses `hmac.compare_digest` to prevent timing attacks.

The only exception is `/api/health` which requires no authentication for monitoring.

## Consequences

- Positive: total simplicity, no cookies, no sessions, no OAuth flows, works with `fetch` without special configuration.
- Negative: credentials sent in plaintext on every request (mitigated by mandatory HTTPS), no logout, no roles/permissions.
- Neutral: frontend triggers native browser login prompt (WWW-Authenticate).

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| OAuth2 + JWT | Industry standard, roles, expiration | Unnecessary complexity for single-user |
| API Key | Simple, no username | No granular revocation, no identity |
| nginx auth_basic | Same simplicity | No programmatic control, no per-route exceptions |

## Risks

If multi-user support is needed, migrate to OAuth2+JWT. The AuthMiddleware can coexist with other authentication schemes.
