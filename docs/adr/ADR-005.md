# ADR-005: Backend Plugin System (Tool-Agnostic Chat)

**Status:** Accepted

**Date:** 2026-06-15

## Context

The chat must support multiple CLI agents (claude, opencode, codex, kimi, etc.) without modifying the HTTP router code. Each backend has a different output format, command-line options, and session identification mechanism.

## Decision

Implement a plugin system via abstract base class `ChatBackend` in `dashboard/backends/protocol.py`. Each backend implements:
- `build_command()` — generates argv for the subprocess.
- `parse_line()` — normalizes a stdout line to a `NormalizedEvent`.
- `validate_available()` — checks if the executable exists in $PATH.

Backends auto-register in `dashboard/backends/__init__.py` and are discovered via `discover()`. Selection is by name via `get(name)` or automatic via `select_backend()`.

## Consequences

- Positive: adding a backend = create a class + register, no routes/ changes, integration with `/api/backends` for dynamic UI.
- Negative: additional abstraction, each backend requires individual maintenance.
- Neutral: normalized event stream (TextDelta, ToolUse, ToolResult, etc.) allows the frontend to render independently of the backend.

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| Shell scripts per backend | Simple, no Python | No typing, no testability, no normalization |
| Declarative JSON config | No code per backend | Limited for complex parsing |
| Separate ADRs per backend | Explicit | Duplication, no polymorphism |

## Risks

Backends may change their output format between versions. Each backend needs regression tests. `parse_line()` must be tolerant of unexpected lines.
