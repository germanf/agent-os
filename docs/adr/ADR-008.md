# ADR-008: Headroom for Context Optimization

**Status:** Proposed

**Date:** 2026-06-20

## Context

CLI agents (claude, opencode) operate with limited context windows. In large projects, context can become saturated with dead code, irrelevant files, and past conversations, reducing response quality.

## Decision

Integrate **Headroom** (https://github.com/nicholasgriffintn/headroom) as a context optimization layer. Headroom provides:
- Project structure analysis to identify relevant files.
- Tree-sitter-based context compression with 60-95% token reduction.
- Automatic CLI command wrapping via `headroom wrap opencode`.

Headroom runs as a proxy between the backend and the CLI agent, analyzing the repository before each invocation and filtering/injecting relevant context.

## Consequences

- Positive: up to 95% fewer tokens per invocation, more focused responses, lower API costs.
- Negative: additional latency per invocation (repository analysis), dependency on Rust toolchain for tree-sitter, possible false negatives in relevant files.
- Neutral: graceful degradation if Headroom is unavailable (system works without it).

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| Manual context prompts | No dependencies | Doesn't scale, error-prone |
| RAG with embeddings | Precise, semantic | Infrastructure complexity (vectors) |
| Native model sliding window | Simple | No granular control, loses old context |

## Risks

Headroom is a smaller project than Hermes. May have less long-term support. Integration must be configurable (on/off per project) to allow selective disabling.
