# ADR-003: SQLite for Chat Persistence

**Status:** Accepted

**Date:** 2026-06-01

## Context

The platform needs to persist project metadata, chats, and messages for the UI (titles, sidebar, grouping). Write concurrency is low (single user). The CLI agent (claude) manages its own conversation state separately.

## Decision

Use SQLite via aiosqlite with WAL mode as the database engine. The schema includes tables for `projects`, `project_folders`, `chats`, and `messages`. Schema migration is idempotent via `ALTER TABLE ADD COLUMN` with try/except.

No PostgreSQL or ORM — SQLite is sufficient for single-user and removes the dependency on a database server.

## Consequences

- Positive: zero configuration, zero infrastructure dependencies, backups = copy one file, WAL mode allows concurrent reads during writes.
- Negative: does not scale horizontally, no native UUID/JSON types (stored as TEXT), no formal migrations (Alembic).
- Neutral: chat IDs are UUID v4 stored as text; timestamps are Unix floats.

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| PostgreSQL | Scalable, native types, migrations | Requires server, more complex in dev |
| JSON file | Simple, portable | No queries, no referential integrity, race conditions |
| Redis | Fast, ephemeral | Not persistent by default, no structured queries |

## Risks

Low. If multi-user support is needed in the future, migration to PostgreSQL is feasible. The current schema is small enough that migration is trivial.
