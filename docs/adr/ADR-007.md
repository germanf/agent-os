# ADR-007: Multi-Agent Architecture with Hermes as Orchestrator

**Status:** Proposed

**Date:** 2026-06-20

## Context

The platform currently runs a single CLI agent per conversation. To scale to multi-agent workflows (planner → developer → reviewer → QA), an orchestrator is needed to coordinate specialized agents, manage shared state, and resolve task dependencies.

## Decision

Integrate **Hermes Agent** (https://github.com/NousResearch/Hermes) as the orchestration layer. Hermes provides:
- Task planning and decomposition into subtasks.
- Routing subtasks to specialized agents (capability-based).
- Shared state management and dependency resolution.
- 25k+ existing tests and 206k stars — avoid building a custom orchestrator.

Executor agents will be OpenCode instances with specialized skills (Planner, Developer, Reviewer, QA), defined via capability-based ABCs.

## Consequences

- Positive: battle-tested architecture, immediate multi-agent capability, clear separation between orchestration and execution.
- Negative: external dependency, non-trivial integration, latency overhead for simple tasks.
- Neutral: system must work without Hermes (single-agent legacy mode). Incremental migration in 4 phases.

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| CrewAI | Mature multi-agent framework | Strong opinion, less flexible than Hermes |
| AutoGen (Microsoft) | Conversational multi-agent | Focused on dialogue, not task execution |
| Custom orchestrator | Full control | High development and maintenance cost |

## Risks

Hermes may have breaking changes. Integration must use an adapter pattern to isolate changes. Phase 2 (OpenCode SDK) must complete before Phase 3 (Hermes).
