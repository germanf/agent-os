# ADR-002: Chat via Subprocess with SSE Streaming

**Status:** Accepted

**Date:** 2026-06-01

## Context

The platform needs to run CLI agents (claude, opencode) as subprocesses and stream their output to the frontend in real time. Full-duplex WebSocket is not required — only unidirectional backend → frontend streaming.

## Decision

Use Server-Sent Events (SSE) over HTTP for log and event streaming. The backend:
1. Creates a subprocess with `asyncio.create_subprocess_exec`.
2. Reads stdout line by line and stores them in a circular buffer.
3. Publishes each line to per-subscriber async queues (fan-out).
4. Serves SSE via FastAPI's `StreamingResponse`.

No WebSocket — the frontend only needs to read, not write over the same channel.

## Consequences

- Positive: simple HTTP, no protocol upgrade, works with nginx without special config (except disabling buffering), native reconnection via EventSource API.
- Negative: ~6 simultaneous connections per domain limit in HTTP/1.1 (mitigated by HTTP/2 in production), no unsolicited server→client messages outside a subscription.
- Neutral: stream includes heartbeats (event: ping) every 25s to keep the connection alive.

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| WebSocket | Bidirectional, no connection limit | More complex, requires upgrade, nginx needs proxy_pass upgrade |
| Long-polling | Universal compatibility | High overhead, not real-time |
| gRPC streaming | Efficient, typed | Complex toolchain, doesn't work behind nginx without special proxy |

## Risks

SSE connections can close silently; the 25s heartbeat mitigates this. The in-memory log buffer is limited to 1 hour TTL and 200 jobs max.
