# ADR-006: Generic Runner with Tool as Free String

**Status:** Accepted

**Date:** 2026-06-01

## Context

The platform runs multiple types of subprocesses (chat with claude, opencode, etc.). Each needs: creation, async execution, stdout capture, cancellation, and log streaming with concurrent subscribers.

## Decision

Implement `dashboard/runner.py` as a generic runner that:
- Creates a `Job` with `id`, `tool` (free string), `command`, `cwd`.
- Executes via `asyncio.create_subprocess_exec`.
- Stores logs in `_log_lines` and publishes to subscriber queues (fan-out).
- Supports cancellation and automatic pruning (1h TTL, max 200 finished jobs).
- Exposes `create_job()`, `run_job()`, `stream_logs()`, `cancel_job()` as public API.

The `tool` field is a free string — the runner does not know about or depend on any specific backend.

## Consequences

- Positive: runner reusable for any CLI tool, decoupled from backends, easy to test.
- Negative: no validation that the command exists before creating the job (validated in the backend plugin).
- Neutral: `stream_logs()` returns raw lines; semantic parsing happens in the backend plugin via `parse_line()`.

## Alternatives Considered

| Alternative | Pros | Cons |
|-------------|------|------|
| Backend-specific inline subprocess | Simple | Duplication, no reuse, no generic cancellation |
| Job queue with Celery | Scalable, distributed | Overkill for single-user, Redis/RabbitMQ dependency |
| Direct asyncio.subprocess in each handler | No abstraction | Repetitive code, no automatic pruning |

## Risks

Child jobs that don't respond to SIGTERM may become orphaned. Current cancellation uses `process.terminate()`. Future iterations may use `process.kill()` with timeout.
